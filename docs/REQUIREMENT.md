# 校园奶茶拼单系统需求文档

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 项目名称 | 校园奶茶拼单系统 |
| 开发模式 | B/S |
| 后端语言 | Python |
| Web 框架 | Flask |
| 数据库 | MySQL 8.0 |
| 数据库连接库 | mysql-connector-python |
| 前端技术 | HTML、Bootstrap、Jinja2 |
| 目标用户 | 校园学生、拼单发起人、系统管理员 |
| 项目目标 | 实现一个小型校园奶茶拼单信息管理系统，并完整覆盖数据库工程作业评分项 |

---

## 2. 项目背景

校园学生经常通过微信群、QQ群、宿舍群进行奶茶拼单，但人工统计订单容易出现以下问题：

1. 饮品库存无法实时控制。
2. 优惠券是否可用难以核验。
3. 拼单完成后金额统计容易出错。
4. 取消拼单时库存、优惠券、订单状态容易不一致。
5. 拼单详情查询不方便。

因此，本系统设计为一个轻量级校园奶茶拼单管理系统，用数据库机制完成核心业务控制，包括：

1. 触发器控制下单合法性。
2. 存储过程完成拼单结算。
3. 事务保证取消拼单的数据一致性。
4. 视图封装拼单详情查询。

### 3.1 业务目标

1. 支持学生参与校园奶茶拼单。
2. 支持拼单发起、下单、完成拼单、取消拼单。
3. 支持饮品库存自动维护。
4. 支持优惠券使用状态自动维护。
5. 支持拼单详情查询和演示。

### 3.2 作业评分目标

1. 能在报告中填写完整的项目信息。
2. 能说明系统配置、数据库连接串及其参数含义。
3. 能展示数据库表设计和关系图。
4. 能演示含事务应用的删除操作。
5. 能演示触发器控制下的添加操作。
6. 能演示存储过程控制下的更新操作。
7. 能演示含视图的查询操作。
8. 能提供可运行的 Python 程序和 MySQL 数据库脚本。

## 5. 业务对象

### 5.1 学生

学生是拼单系统中的购买者。

核心属性：

1. 学生编号。
2. 姓名。
3. 手机号。
4. 状态。

学生状态包括：

| 状态 | 含义 |
| --- | --- |
| ACTIVE | 正常，可以下单 |
| LOCKED | 被限制，不能下单 |

---

### 5.2 奶茶店

奶茶店提供饮品。

核心属性：

1. 店铺名称。
2. 校区位置。
3. 联系电话。
4. 状态。

店铺状态包括：

| 状态 | 含义 |
| --- | --- |
| OPEN | 营业中 |
| CLOSED | 停业 |

---

### 5.3 饮品

饮品属于某个奶茶店。

核心属性：

1. 饮品名称。
2. 所属店铺。
3. 单价。
4. 库存数量。
5. 上架状态。

饮品状态包括：

| 状态 | 含义 |
| --- | --- |
| ON_SALE | 可购买 |
| OFF_SALE | 已下架 |

---

### 5.4 拼单

拼单表示一次集体购买活动。

核心属性：

1. 拼单标题。
2. 所属奶茶店。
3. 发起人。
4. 截止时间。
5. 拼单状态。
6. 拼单总金额。

拼单状态包括：

| 状态 | 含义 |
| --- | --- |
| OPEN | 进行中，可加入 |
| FINISHED | 已完成，不可继续加入 |
| CANCELED | 已取消 |

---

### 5.5 订单明细

订单明细表示某个学生在某个拼单中购买了某个饮品。

核心属性：

1. 拼单编号。
2. 学生编号。
3. 饮品编号。
4. 数量。
5. 优惠券编号。
6. 原价金额。
7. 优惠金额。
8. 实付金额。

订单明细状态包括：

| 状态 | 含义 |
| --- | --- |
| CREATED | 已创建，拼单未结算 |
| PAID | 已结算 |
| CANCELED | 已取消 |

---

### 5.6 优惠券

优惠券属于学生。

核心属性：

1. 所属学生。
2. 优惠金额。
3. 最低使用金额。
4. 有效期。
5. 使用状态。

优惠券状态包括：

| 状态 | 含义 |
| --- | --- |
| UNUSED | 未使用 |
| USED | 已使用 |

---

### 5.7 操作日志

操作日志记录关键数据库操作。

核心属性：

1. 操作类型。
2. 操作说明。
3. 操作时间。

# 7. P0 需求

## 7.1 P0 数据库需求

### P0-DB-001：系统必须使用 MySQL 数据库

系统必须使用 MySQL 作为后台数据库，不得使用 SQLite、Access 等桌面数据库。

验收标准：

1. 数据库运行在 MySQL 8.0 或兼容版本。
2. 程序通过 MySQL 连接库访问数据库。
3. 报告中能够说明数据库版本和连接方式。

---

### P0-DB-002：系统必须创建独立数据库

系统必须创建名为 `milk_tea_group_db` 的数据库。

验收标准：

1. 执行建库脚本后存在 `milk_tea_group_db`。
2. 所有业务表、视图、触发器、存储过程均位于该数据库中。

---

### P0-DB-003：系统必须包含学生表

系统必须创建 `students` 表，用于存储学生信息。

字段要求：

| 字段 | 说明 |
| --- | --- |
| student_id | 学生主键 |
| student_no | 学号 |
| name | 姓名 |
| phone | 手机号 |
| status | 学生状态 |
| created_at | 创建时间 |

验收标准：

1. `student_id` 为主键。
2. `student_no` 唯一。
3. `status` 支持 `ACTIVE` 和 `LOCKED`。
4. 至少初始化 3 条学生数据。

---

### P0-DB-004：系统必须包含奶茶店表

系统必须创建 `shops` 表，用于存储奶茶店信息。

字段要求：

| 字段 | 说明 |
| --- | --- |
| shop_id | 店铺主键 |
| shop_name | 店铺名称 |
| campus_location | 校区位置 |
| phone | 联系电话 |
| status | 店铺状态 |

验收标准：

1. `shop_id` 为主键。
2. `shop_name` 不为空。
3. `status` 支持 `OPEN` 和 `CLOSED`。
4. 至少初始化 2 条店铺数据。

---

### P0-DB-005：系统必须包含饮品表

系统必须创建 `drinks` 表，用于存储饮品信息。

字段要求：

| 字段 | 说明 |
| --- | --- |
| drink_id | 饮品主键 |
| shop_id | 所属店铺 |
| drink_name | 饮品名称 |
| price | 单价 |
| stock | 库存 |
| status | 饮品状态 |

验收标准：

1. `drink_id` 为主键。
2. `shop_id` 外键引用 `shops(shop_id)`。
3. `price` 必须大于 0。
4. `stock` 必须大于等于 0。
5. `status` 支持 `ON_SALE` 和 `OFF_SALE`。
6. 至少初始化 5 条饮品数据。

---

### P0-DB-006：系统必须包含拼单表

系统必须创建 `group_orders` 表，用于存储拼单主信息。

字段要求：

| 字段 | 说明 |
| --- | --- |
| group_order_id | 拼单主键 |
| shop_id | 奶茶店编号 |
| creator_student_id | 发起学生编号 |
| title | 拼单标题 |
| deadline_at | 拼单截止时间 |
| status | 拼单状态 |
| total_amount | 拼单总金额 |
| created_at | 创建时间 |

验收标准：

1. `group_order_id` 为主键。
2. `shop_id` 外键引用 `shops(shop_id)`。
3. `creator_student_id` 外键引用 `students(student_id)`。
4. `status` 支持 `OPEN`、`FINISHED`、`CANCELED`。
5. `total_amount` 默认值为 0。
6. 至少初始化 2 条拼单数据。

---

### P0-DB-007：系统必须包含订单明细表

系统必须创建 `order_items` 表，用于存储学生参与拼单的订单明细。

字段要求：

| 字段 | 说明 |
| --- | --- |
| order_item_id | 订单明细主键 |
| group_order_id | 拼单编号 |
| student_id | 学生编号 |

| coupon_id | 优惠券编号 |
| quantity | 购买数量 |
| item_amount | 原价金额 |
| discount_amount | 优惠金额 |
| pay_amount | 实付金额 |
| status | 明细状态 |
| created_at | 创建时间 |

验收标准：

1. `order_item_id` 为主键。
2. `group_order_id` 外键引用 `group_orders(group_order_id)`。
3. `student_id` 外键引用 `students(student_id)`。
4. `drink_id` 外键引用 `drinks(drink_id)`。
5. `coupon_id` 可为空。
6. `quantity` 必须大于 0。
7. `status` 支持 `CREATED`、`PAID`、`CANCELED`。

---

字段要求：

4. `min_order_amount` 必须大于等于 0。

字段要求：

## 7.4 P0 触发器添加需求

### P0-TRG-001：系统必须创建订单插入前触发器

系统必须创建 `trg_before_order_item_insert` 触发器。

验收标准：

1. 触发器类型为 `BEFORE INSERT`。
2. 触发器作用于 `order_items` 表。
3. 触发器源码能截图放入报告。

---

### P0-TRG-002：触发器必须检查学生状态

插入订单明细前，触发器必须检查学生是否存在且状态为 `ACTIVE`。

验收标准：

1. 学生不存在时，插入失败。
2. 学生状态为 `LOCKED` 时，插入失败。
3. 失败时数据库返回明确错误信息。

---

### P0-TRG-003：触发器必须检查拼单状态

插入订单明细前，触发器必须检查拼单是否存在且状态为 `OPEN`。

验收标准：

1. 拼单不存在时，插入失败。
2. 拼单状态为 `FINISHED` 时，插入失败。
3. 拼单状态为 `CANCELED` 时，插入失败。

---

### P0-TRG-004：触发器必须检查饮品状态

插入订单明细前，触发器必须检查饮品是否存在且状态为 `ON_SALE`。

验收标准：

1. 饮品不存在时，插入失败。
2. 饮品状态为 `OFF_SALE` 时，插入失败。
3. 失败信息可在页面显示。

---

### P0-TRG-005：触发器必须检查饮品是否属于拼单店铺

插入订单明细前，触发器必须检查饮品所属店铺与拼单所属店铺一致。

验收标准：

1. 若饮品不属于该拼单店铺，插入失败。
2. 若饮品属于该拼单店铺，继续执行后续检查。

---

### P0-TRG-006：触发器必须检查购买数量

插入订单明细前，触发器必须检查购买数量大于 0。

验收标准：
