# P2 分 Stage 搭建计划

本文档用于指导“校园奶茶拼单系统”P2 范围的逐 stage 实施。P2 只在 P0/P1 已可运行、可演示的基础上提升产品化程度；每个 stage 都必须形成 `plan -> develop -> verify -> commit` 闭环。

## 0. 执行约定

1. 开始 P2 前必须先确认 P0/P1 核心流程仍可运行：加入拼单、完成拼单、取消拼单、拼单详情查询、基础数据查看、操作日志查看、首页统计。
2. 每个 stage 开始前，先确认本 stage 的目标、覆盖的 P2 条目、涉及文件、依赖的 P0/P1 能力和验收方式。
3. 每个 stage 只实现当前范围，不提前实现后续 stage；认证、角色、创建拼单、过期关闭、备注、排行和导出必须分阶段落地。
4. Commit title 使用 `<action>: <message>` 格式；`action` 优先使用 `update`、`fix`、`delete`、`doc`。
5. 每个 commit 除标题外必须包含详细描述正文，说明本次变更内容、验证结果和影响范围。
6. 文档、配置示例和 README 不写入本机绝对路径、真实用户名、设备信息或真实数据库密码。
7. 运行 Python 或 Flask 前先激活项目约定的 conda 环境；依赖变更必须同步更新 `requirements.txt`。
8. Python SQL 操作必须使用参数化查询；事务失败必须 rollback；数据库连接和 cursor 必须关闭。
9. P2 允许按需求新增最小数据库结构，例如登录账号表和订单备注字段；所有结构变更必须写入 `schema.sql`，并配套稳定演示数据。
10. P2 登录账号仅用于课程演示，不写入真实账号、真实密码或个人身份信息；若需要密码，使用合成演示账号和安全哈希。
11. P2 权限实现以 Flask session 和页面级访问控制为主，不引入复杂第三方认证框架。
12. P2 自动关闭过期拼单采用请求时检查或显式服务函数触发，不要求后台定时任务或操作系统级计划任务。
13. 本计划只覆盖 `docs/REQUIREMENT.md` 第 9 章有验收标准的 P2 条目；二维码和移动端适配暂不纳入本轮 P2 实施边界。

## 1. P2 范围摘要

P2 交付目标是让系统从数据库演示应用进一步接近可用的产品原型：

1. 提供简单登录页面，支持登录成功、失败提示和退出登录。
2. 区分学生和管理员角色，按角色展示入口并限制关键页面。
3. 学生用户只能参与拼单，并查询自己的订单。
4. 管理员可以完成拼单、取消拼单、查看全部数据和管理演示流程。
5. 提供创建拼单页面，创建后拼单状态为 `OPEN`。
6. 支持过期拼单自动关闭，过期拼单不能继续加入。
7. 支持学生下单备注，并在拼单详情中展示备注。
8. 提供饮品销量排行和学生消费排行。
9. 支持将拼单详情查询结果导出为 CSV。
10. 更新演示说明、验收清单和运行文档，确保 P2 不破坏 P0/P1 评分操作。

建议 P2 新增或增强页面路径：

| 页面名称 | 路径 | 覆盖需求 |
| --- | --- | --- |
| 登录 | `/login` | P2-AUTH-001 |
| 退出登录 | `/logout` | P2-AUTH-001 |
| 创建拼单 | `/group/create` | P2-ORDER-001 |
| 饮品销量排行 | `/stats/drinks` | P2-CHART-001 |
| 学生消费排行 | `/stats/students` | P2-CHART-002 |
| 拼单详情 CSV 导出 | `/group/query/export` | P2-EXPORT-001 |

## 2. Stage Gate 模板

每个 stage 必须按以下格式执行：

| 步骤 | 要求 |
| --- | --- |
| Plan | 明确本 stage 目标、覆盖的 P2 编号、改动文件、依赖关系、风险和验收命令。 |
| Develop | 只实现本 stage 范围内的文件改动，不提前实现后续 stage。 |
| Verify | 执行本 stage 的最小可验证检查，并记录结果；涉及数据库时优先从干净数据库重放。 |
| Commit | 验证通过后提交；提交标题使用 `<action>: <message>`；提交正文写清变更内容、验证结果和影响范围；提交后确认工作区只剩用户允许保留的变更。 |

Commit 正文建议模板：

```text
Changes:
- <本 stage 的主要变更>

Verification:
- <已执行的验证步骤和结果>

Scope:
- <影响范围或未覆盖事项>
```

### Stage 07：过期拼单自动关闭

Develop：

1. 更新 `schema.sql`，如采用 `EXPIRED`，同步更新 `group_orders.status` 检查约束、种子数据、视图查询状态选项。

3. 在首页、加入拼单、拼单查询、完成拼单前调用过期关闭逻辑，或在统一请求钩子中调用。
4. 更新订单插入触发器，确保过期或非 `OPEN` 拼单不能继续加入。
5. 可为过期关闭写入 `operation_logs`，日志类型建议为 `EXPIRE_GROUP_ORDER`。

Verify：

2. 访问触发过期检查的页面，确认该拼单状态变为 `EXPIRED` 或 `CANCELED`。

4. 确认未过期 `OPEN` 拼单仍可正常加入。
5. 确认完成拼单不会完成已经过期关闭的拼单。

Commit：

1. 提交标题：`update: expire overdue group orders`。

## 5. 后续执行方式

后续正式开发时，每次只启动一个 stage。每个 stage 的工作完成后，必须先完成 verify，再提交 commit，最后再进入下一 stage。

建议每个 stage 完成后记录：

1. 本 stage 实际改动文件。
2. 实际执行的验证命令或手工验证步骤。
3. 验证结果。
4. 提交哈希。
5. 遗留问题或下一 stage 依赖。
